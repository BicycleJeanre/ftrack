# Account Groups + Compound Transactions + Capital/Interest Ledger (Combined Plan)

This document merges:

- `Documentation/TECH_ACCOUNT_GROUPS_AND_ROLLUPS.md` (groups/rollups + compound transactions)
- `capital-interest-subtotal-plan.md` (capital vs interest split + aggregation utility)

It also clarifies how **periodic change / interest accrual** fits the model given the requirement:

> Interest must appear in an **interest expense** account (i.e., not only as an internal “interest delta” on a balance).

---

## 1.0 Goals (what this combined model must support)

1. **Account Groups (tree) for rollups/navigation** without “posting to parent” ambiguity.
2. **Split payments** as one user intent that expands to multiple 2‑leg transactions (compatible with current schema/UI).
3. **Capital vs interest everywhere** (transactions, projections, toolbar totals, summary cards) via a single Aggregation Utility.
4. **Periodic change / interest accrual** produces **ledger-visible postings** into an interest account, not just an internal balance adjustment.

---

## 2.0 Core concept mapping

### 2.1 Account Groups (rollups, not postable)

Same as the groups/rollups plan:

- Groups are computed rollups; they are **not** selectable as `primaryAccountId`.
- Rollup totals are derived by summing descendant member accounts.

### 2.2 Compound Transactions (split payments)

Same as the groups/rollups plan:

- A “compound transaction” is stored as **N normal canonical transactions** linked by a shared `transactionGroupId`.
- Optional `transactionGroupRole` identifies the component semantics (e.g. `principal`, `interest`, `fee`).

This is the key bridge for capital/interest: compound components can be classified deterministically.

### 2.3 Capital vs Interest classification (single source of truth)

We want downstream UIs to *sum*, not *re-derive*, capital/interest splits. The most robust place to decide the split is on **projection occurrences** (and/or a shared “occurrence normalizer” used by both projections + UI lists).

Minimal rule set:

- Default: `capitalAmount = amount`, `interestAmount = 0`.
- If `transactionGroupRole === 'interest'`: `capitalAmount = 0`, `interestAmount = amount`.
- If an occurrence is generated by an interest accrual rule: `capitalAmount = 0`, `interestAmount = amount`.

## 3.0 Periodic Change: from “balance adjustment” to “ledger posting”

### 3.1 Why storing “interest as another recurring transaction” is not enough by itself

A normal recurring transaction in FTrack has an amount that is:

- constant, or
- escalated vs time via `transaction.periodicChange` (value changes as a function of elapsed time).

But **interest accrual** needs an amount that is a function of **balance at a point in time**:

- `interest = f(currentBalance, rate, timeSlice)`

That depends on the account’s evolving balance (driven by other transactions), so it cannot be represented as a purely time-based escalation on a standalone recurring transaction without making the engine circular or inaccurate.

### 3.2 The combined model’s approach: store the *rule*, generate the *posting occurrences*

Keep `account.periodicChange` / `account.periodicChangeSchedule` as the user-facing “rate rule”, but change how projections treat it:

- Instead of directly mutating `currentBalance` and emitting an `interest` delta on the projection row,
- **generate derived transaction occurrences** per period (or per schedule segment) that *post* interest into a configured interest account.

This achieves:

- Interest becomes visible as ledger activity (filters, rollups, compound grouping, totals).
- No inflation of `scenario.transactions` storage: derived occurrences live in the **projection bundle**, not in scenario storage.

### 3.3 Required configuration to know *where* interest posts

For each postable account that has periodic change, add (or derive) a “posting target” configuration:

- `account.interestAccountId` (where interest is categorized)
- `account.interestPostingDirection` (how to wire primary/secondary + transactionTypeId)

Default wiring for loans (accruing interest increases liability and increases interest expense):

- Derived interest occurrence:
  - `primaryAccountId = principalAccountId` (the balance being charged)
  - `secondaryAccountId = interestExpenseAccountId`
  - `transactionTypeId = expense`
  - `amount = computedInterestForSlice`

This produces the desired behavior under the existing projection transaction application rules:

- The principal/liability balance moves “more negative” (more debt).
- The interest expense account accumulates the interest as spend/category.

Other account types (savings/investments) can be added later with different wiring (interest income vs expense).

### 3.4 How to compute the derived interest postings (engine-level)

During projection period calculation:

1. Apply “normal” transaction occurrences for the period (existing behavior).
2. Determine active periodic change for the period (existing schedule logic).
3. Compute the interest delta for the slice:
   - Use the existing periodic-change math (`calculatePeriodicChange`) to compute:
     - `after = calculatePeriodicChange(before, periodicChange, yearsInSlice)`
     - `delta = after - before`
   - Convert to a posting amount:
     - For loans, interest expense should be **positive** spend, so `postingAmount = Math.abs(delta)` when the direction indicates a charge.
4. Apply the derived interest transaction occurrence using the same “2-legged” transfer logic so it affects both accounts.
5. Mark the occurrence as interest for aggregation:
   - `capitalAmount = 0`, `interestAmount = postingAmount`
   - `source = 'derivedPeriodicChange'` (internal tag for UI/debugging)

This keeps projection correctness consistent: interest is computed from the actual evolving balance, and also becomes ledger-visible.

## 4.0 Aggregation Utility (capital/interest totals everywhere)

Adopt the capital-interest plan’s “utility-first” approach, but standardize the input record format:

Any record that can be aggregated (transaction rows, budget rows, projection occurrences) should expose:

- `capitalAmount`
- `interestAmount`
- `direction` (in/out relative to a chosen account perspective) OR enough canonical fields to derive it consistently

The utility returns:

- `capitalIn`, `capitalOut`, `interestIn`, `interestOut`, `total` (and optional breakdown/description)

Group rollups become straightforward:

- Expand a group to member account IDs.
- Aggregate over the union of those accounts’ rows.

## 5.0 Phased implementation plan (combined)

### Phase 1 — Plumbing: capital/interest fields + Aggregation Utility

1. Define canonical “aggregatable row” shape for:
   - transactions list rows
   - budget rows
   - projection occurrences/rows
2. Implement Aggregation Utility that consumes that shape and outputs capital/interest totals.
3. Update existing totals surfaces (toolbar totals, summary cards, projections grid totals) to use the utility.

### Phase 2 — Account Groups (rollups)

1. Add `scenario.accountGroups` schema + CRUD.
2. Add group membership resolution utility (groupId → accountId set).
3. Add rollup totals in summary/totals panels by summing member accounts (via Aggregation Utility).

### Phase 3 — Compound Transactions (split payments)

1. Add `transactionGroupId` + optional `transactionGroupRole`.
2. UI creation flow to create/edit a compound payment (creates N canonical transactions).
3. Optional UI grouping/expand-collapse by `transactionGroupId`.
4. Capital/interest classification:
   - interest components flagged by `transactionGroupRole`.

### Phase 4 — Periodic Change postings (interest must be ledger-visible)

1. Add `interestAccountId` (and any minimal direction/config) to accounts that use periodic change.
2. Update projection generation to **generate and apply derived interest occurrences** instead of (or in addition to) internal `periodInterest` deltas.
3. Ensure derived occurrences feed the Aggregation Utility as interest-only amounts.

### Phase 5 — Top-down loan allocations (optional but synergistic)

1. For “Loan payment” templates, compute interest component from projected principal balance + rate rule for that payment date.
2. Allocate payment total via waterfall (fees → interest → principal).
3. Store as compound payment components so the ledger matches the split and all totals roll up naturally.

## 6.0 Notes on “transaction count inflation”

- The combined plan does **not** require storing one new canonical transaction per interest event in `scenario.transactions`.
- It stores the **interest rule** (periodic change + posting target) on the account, and generates occurrences during projection runs.
- The projection bundle already stores per-period rows; adding derived “interest postings” to the projection layer is a bounded increase that scales with projection periods, not with user-edited scenario storage.

If we later want to show derived interest rows in the transactions grid, we can:

- show them only in the Projections view, or
- add an “Materialize derived postings” action that writes them into scenario transactions for a selected date range (opt-in).
